# AIRS：AI Interface Rendering Specification

**AIRS 是面向 AI 原生操作系统的 AI 界面渲染协议与交互范式。**

AIRS 用于规范 AI 在自主生成和组织界面时，如何以结构化 `Intent View` 的形式描述用户意图、任务阶段、服务数据、界面组件、语义动作、权限要求和安全约束，并由操作系统运行时负责校验、渲染、交互映射和安全执行。

AIRS 的目标不是让 AI 随意生成页面，也不是让大模型直接编写和执行前端代码，而是为“AI 自主生成界面”这件事建立一套操作系统级的协议和范式。

> 当前状态：AIRS 仍是一个面向讨论的早期概念草案，不是已经定稿的标准。

## AIRS 要解决什么问题

如果未来 AI 不只是回答文本，而是会根据用户意图调用服务、组织数据、生成界面并推动任务执行，那么“AI 生成界面”就不能只是自由输出。它需要被约束、校验、审计，并能够适配不同设备。

没有协议约束时，会出现几个问题：

- AI 可能生成任意 UI 代码，运行时难以安全校验。
- 服务数据、模型推理内容、界面展示内容之间缺少清晰来源。
- 用户点击、确认、授权等行为缺少统一语义。
- 高风险动作可能在没有明确权限和确认的情况下被执行。
- 同一个任务界面难以在手机、桌面、车机、可穿戴设备或语音设备上稳定适配。

AIRS 的核心思路是：AI 不直接输出任意前端代码，而是输出结构化 `Intent View`；系统运行时负责验证、渲染、适配、权限控制和动作执行。

## 核心流程

```text
用户输入
语音 / 图片 / 文本 / 视频 / 上下文
        |
        v
AI Agent
意图理解 / 任务规划 / 服务调用
        |
        v
AI 原生服务
业务数据 + 渲染提示 + 风险声明
        |
        v
AIRS Intent View
意图 / 阶段 / 组件 / 动作 / 安全约束
        |
        v
AIRS Runtime
校验 / 权限控制 / 设备适配 / 渲染
        |
        v
用户交互
触摸 / 语音 / 手势 / 确认
        |
        v
语义动作反馈
```

## 关键概念

- `Intent View`：AI 生成界面的结构化描述。
- `Task Stage`：当前任务所处阶段，例如澄清、搜索、选择、确认、执行、结果或错误恢复。
- `Render Component`：协议层面的界面组件，例如摘要卡片、选项卡片、表单、媒体视图或权限提示。
- `Semantic Action`：具有明确语义、风险等级和后续阶段的用户动作。
- `Data Source`：界面所展示数据的来源声明。
- `Safety Gate`：确认、权限、禁止自动执行、审计等安全规则。
- `AIRS Runtime`：负责校验和渲染 `Intent View` 的系统运行时。

## 第一版范围

AIRS 0.1 先关注最小可讨论协议形状：

- 最小 `Intent View` 结构
- 通用任务阶段
- 核心渲染组件类型
- 语义动作定义
- 数据来源声明
- 风险等级和权限门槛
- 几个典型场景示例

## 参与讨论

这个仓库不是为了宣布一个已经完成的标准，而是为了公开一个可以被讨论、质疑、修改和演化的协议草案。

可以从这些问题开始：

- AI 原生操作系统运行时在渲染 AIRS 视图时应该承担哪些职责？
- 一个最小可用的 `Intent View` 应该包含哪些字段？
- AIRS 0.1 应该标准化哪些组件？
- 风险等级、权限和用户确认应该如何表达？
- AIRS Runtime 应该解释、编译，还是转换 `Intent View`？

贡献方式见 [CONTRIBUTING.md](CONTRIBUTING.md)。
